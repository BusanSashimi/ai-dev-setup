# 프로젝트 폴더 구조 가이드라인

## 개요

본 문서는 개발팀을 위한 프로젝트 폴더 구조 및 네이밍 컨벤션을 정의합니다. 일관된 구조를 통해 코드의 가독성과 유지보수성을 향상시키고, 팀원 간 협업 효율을 높이는 것을 목표로 합니다.

## 핵심 원칙

1. 모듈 기반 설계: 데이터베이스 테이블부터 프론트엔드까지 모듈 중심으로 구성
2. 폴더 구조의 일관성: 백엔드와 프론트엔드에서 동일한 모듈 구조 유지
3. 유연한 UI 구성: 프론트엔드 메뉴 구조와 폴더 구조는 일치시키지 않음
4. 모듈은 1차, 2차 depth로 규정

## 데이터베이스 테이블 네이밍 컨벤션

- 모든 테이블은 모듈 기반으로 설계하며 접두어를 사용하여 소속 모듈을 명시
- 네이밍 패턴: `[프로젝트명]_[모듈명]_[세부기능]`
- 예시:
  ```
  itax_board                  # 게시판 기본 테이블
  itax_board_email            # 게시판 이메일 관련 테이블
  itax_board_hist             # 게시판 히스토리 테이블
  itax_board_sms              # 게시판 SMS 관련 테이블

  itax-shared-email           # 공유 이메일 테이블
  itax-shared-email-receive   # 공유 이메일 수신 테이블
  itax-shared-sms             # 공유 SMS 테이블
  itax-shared-sms-receive     # 공유 SMS 수신 테이블
  itax-shared-sms-locker      # 공유 SMS 보관함 테이블
  itax-shared-tel             # 공유 전화 테이블
  itax-shared-tel-caller      # 공유 전화 발신자 테이블

  itax_singo_additional       # 신고 부가 정보 테이블
  itax_singo_corporate        # 신고 법인 관련 테이블
  itax_singo_income           # 신고 소득 관련 테이블
  itax_singo_income_human     # 신고 소득 인적 정보 테이블
  itax_singo_income_rental    # 신고 소득 임대 정보 테이블
  itax_singo_pay              # 신고 납부 관련 테이블
  itax_singo_memo             # 신고 메모 테이블
  ```

## 백엔드 프로젝트 폴더 구조

### 기본 원칙

- 기능 중심으로 폴더 구조 정리
- `modules` 폴더 아래에 모든 모듈 정의
- 각 모듈은 표준 구성 요소를 포함: controller, service, dao, type

### 모듈 구조

1. **1차 depth 모듈** (단일 모듈)

   ```
   modules/
   └── board/
       ├── controller/
       ├── service/
       ├── dao/
       └── type/
   ```

2. **2차 depth 모듈** (복합 모듈)

   ```
   modules/
   ├── shared/
   │   ├── email/
   │   │   ├── controller/
   │   │   ├── service/
   │   │   ├── dao/
   │   │   └── type/
   │   ├── sms/
   │   │   ├── controller/
   │   │   ├── service/
   │   │   ├── dao/
   │   │   └── type/
   │   └── tel/
   │       ├── controller/
   │       ├── service/
   │       ├── dao/
   │       └── type/
   │
   └── singo/
       ├── additional/
       │   ├── controller/
       │   ├── service/
       │   ├── dao/
       │   └── type/
       ├── corporate/
       │   ├── controller/
       │   ├── service/
       │   ├── dao/
       │   └── type/
       ├── income/
       │   ├── controller/
       │   ├── service/
       │   ├── dao/
       │   └── type/
       ├── pay/
       │   ├── controller/
       │   ├── service/
       │   ├── dao/
       │   └── type/
       └── memo/
           ├── controller/
           ├── service/
           ├── dao/
           └── type/
   ```

## 프론트엔드 프로젝트 폴더 구조

### 기본 원칙

- 기능 중심으로 폴더 구조 정리
- `modules` 폴더 아래에 모든 모듈 정의
- 각 모듈은 표준 구성 요소를 포함: pages, store, type
- 화면 구성과 폴더 구조는 일치시키지 않음 (유연한 UI 구성 가능)

### 모듈 구조

1. **1차 depth 모듈** (단일 모듈)

   ```
   modules/
   └── board/
       ├── pages/
       │   ├── hist/
       │   │   └── list.vue
       │   ├── sms/
       │   │   └── list.vue
       │   ├── email/
       │   │   └── list.vue
       │   ├── list.vue
       │   ├── list-search.vue
       │   └── list-table.vue
       ├── store/
       │   └── board.store
       └── type/
           └── board.type
   ```

2. **2차 depth 모듈** (복합 모듈)

   ```
   modules/
   ├── shared/
   │   ├── email/
   │   │   ├── pages/
   │   │   ├── store/
   │   │   └── type/
   │   ├── sms/
   │   │   ├── pages/
   │   │   ├── store/
   │   │   └── type/
   │   └── tel/
   │       ├── pages/
   │       ├── store/
   │       └── type/
   │
   └── singo/
       ├── additional/
       │   ├── pages/
       │   ├── store/
       │   └── type/
       ├── corporate/
       │   ├── pages/
       │   ├── store/
       │   └── type/
       ├── income/
       │   ├── pages/
       │   ├── store/
       │   └── type/
       ├── pay/
       │   ├── pages/
       │   ├── store/
       │   └── type/
       └── memo/  # 화면은 없고 store만 규정, 각 세목에서 import하여 사용
           ├── store/
           └── type/
   ```

## 추가 가이드라인

1. **모듈 정의의 일관성**: 백엔드와 프론트엔드에서 동일한 모듈 구조를 유지하여 개발 일관성 확보
2. **UI와 폴더 구조의 분리**: 프론트엔드 메뉴 구조는 폴더 구조와 일치시키지 않아도 됨 (유연한 UI 구성 가능)
3. **공유 모듈 활용**: memo와 같은 공유 모듈은 필요에 따라 다른 모듈에서 import하여 사용
4. **표준 구성 요소**: 각 모듈은 정해진 표준 구성 요소를 포함해야 함

## 결론

본 가이드라인은 프로젝트의 일관성과 유지보수성을 높이기 위한 기본 틀을 제공합니다. 팀원들은 이 가이드라인을 따라 개발하여 효율적인 협업과 코드 관리가 가능하도록 해주시기 바랍니다.
