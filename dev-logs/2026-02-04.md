# Dev Log - 2026-02-04

## Summary

Today's work focused on implementing and debugging a **frontend composite recording system** for the ASMR recorder's 4-section preview layout. This enables WYSIWYG (What You See Is What You Get) recording where the frontend composites multiple video sources and sends frames to the backend for encoding.

### Key Accomplishments
- Implemented full frontend-to-backend frame pipeline
- Fixed multiple React render loop bugs

## Timeline

### 1. [19:24] Region Selector Enhancement
Added live video feed display to RegionSelector component for screen capture region selection, so users can see actual screen content while selecting a capture region.

### 2. [19:43] Core Recording Architecture Implementation
Implemented frontend composite recording system for 4-section preview layout.

**Backend Changes:**
- Added `ExternalRecordingConfig` type to Rust backend (`recording.rs`)
- Created `external_recorder.rs` module (~420 lines) for receiving frames from frontend
  - Frame reception and validation
  - Audio capture (mic + system audio)
  - Audio mixing
  - Video encoding via FFmpeg
- Added Tauri commands: `start_external_recording`, `receive_video_frame`

**Frontend Changes:**
- Created `RecordingCanvas` component for compositing sections and capturing frames
  - Composites 4 video/canvas sources into a 2x2 grid
  - Captures frames at target FPS
  - Sends frames to Rust backend via IPC
- Updated `RecordingContext` with external recording mode support
- Updated `Toolbar` to use external recording for section-based preview

### 3. [20:00] Performance Optimization Round 1
**Problem:** Tauri IPC with JSON serialization was too slow for real-time video (full frames ~2MB each).

**Solutions Applied:**
- Added frame scaling (1/6 resolution) to reduce data transfer overhead
- Added frame dropping mechanism with backpressure (`isSendingRef`)
- Added logging for frame capture progress
- Reduced default frame rate from 30fps to 15fps
- Note: Current limitation requires low resolution due to IPC throughput constraints

### 4. [20:30] Debug Logging Enhancement
**Added comprehensive logging to diagnose recording issues:**
- Section source states (video readyState, canvas dimensions)
- Interval ID tracking to monitor recording loop health
- Periodic logging (every 50 frames) to track long-running issues
- Increased resolution scale from 1/6 to 1/2 for readable text (960x540)
- Investigating: camera section showing black, recording stopping early

### 5. [20:45] Bug Fix: Recording Interval Restarting
**Root Cause:** `sectionSources` dependency in `compositeFrame` callback caused the callback to be recreated on every render.

**Effect:** Recording interval restarted every ~5 seconds, explaining 71-frame limit.

**Solution:**
- Used ref to access sectionSources (`sectionSourcesRef`)
- Removed from callback dependency array
- Added watchdog timer to detect frame sending stalls
- Added useEffect logging to track interval lifecycle
- Improved error handling for "Not recording" race condition on stop

### 6. [21:00] Bug Fix: Component Still Restarting
**Root Cause:** useEffect dependencies included changing callbacks (`startFrameCapture`, `stopFrameCapture`, `captureAndSendFrame`).

**Effect:** Hundreds of restarts over 88 seconds, zero frames captured.

**Solution:**
- Inlined all logic directly in useEffect, only depend on `isRecording`
- Used refs for all dynamic values (dimensions, frameRate, callbacks)
- Removed preview rendering loop (not needed for hidden canvas)
- Reduced resolution from 1/2 to 1/4 scale (480x270) due to IPC bottleneck
- Testing showed 1/2 scale causes 79% frame drop rate (186/234 frames dropped)

### 7. [21:07] Bug Fix: Video Recording Stopping After ~10 Seconds
**Root Cause 1:** React infinite render loop in `preview.tsx`
- `setVideoRefVersion(v => v + 1)` called inside video element ref callback
- Caused "Maximum update depth exceeded" error and UI freezes

**Root Cause 2:** Slow JSON serialization
- `Array.from(imageData.data)` on 518,400 elements took 100-500ms per frame
- At 15fps (66ms interval), most frames were dropped

**Fix 1 - React Loop (`preview.tsx`):**
```javascript
// Before: State-based approach causing loops
const [videoRefVersion, setVideoRefVersion] = useState(0);
ref={(el) => {
  videoRefs.current[index] = el;
  if (el && !processedVideoRefs.current.has(el)) {
    setVideoRefVersion(v => v + 1); // PROBLEM
  }
}}

// After: Getter function approach
const getSectionSources = useCallback(() => { ... }, [sectionState.sections]);
// No setState in ref callbacks
```

**Fix 2 - Base64 Encoding (`recording-canvas.tsx`, `lib.rs`):**
```javascript
// Before: Slow JSON array (518,400 numbers)
const dataArray = Array.from(imageData.data);
await invoke("receive_video_frame", { data: dataArray, ... });

// After: Fast base64 string (~691KB string)
const base64Data = btoa(String.fromCharCode.apply(null, chunks));
await invoke("receive_video_frame_base64", { dataBase64: base64Data, ... });
```

**New Rust command added:**
```rust
fn receive_video_frame_base64(data_base64: String, ...) -> Result<(), String> {
    let data = STANDARD.decode(&data_base64)?;
    recorder.receive_frame(data, width, height, timestamp_ms)
}
```

**New dependency:** `base64 = "0.22"` in `Cargo.toml`

- Recording resolution limited to 480x270 (1/4 scale) due to IPC throughput
- Tauri 2.0 lacks native binary IPC, requiring base64 workaround
- Frame rate limited to 15fps for reliable capture
